Frequently Asked Questions

Everything you wanted to know, answered.

No — never. Needle runs entirely on your machine. The index is built locally and stored in ~/.needle/index/. The web UI is served by a local process on localhost:7700. No file content, metadata, or telemetry is sent anywhere.

The only time network traffic occurs is if you use the Ask feature and have an API key configured for an external LLM (Claude, OpenAI, Groq). In that case, only your question and the retrieved code snippets are sent to that provider — and only when you click Ask.

Hybrid search combines two independent retrieval methods and merges their results:

  • BM25 (keyword) — exact token matching. Great for finding a specific function name, a variable, or a unique error string.
  • HNSW vector search (semantic) — embedding-based similarity. Great for conceptual queries like "retry logic", "auth middleware", or "parse JSON payload".

Results are merged with Reciprocal Rank Fusion (RRF), which rewards chunks that score highly in both methods. A chunk that's a perfect keyword match AND semantically close will rank #1 every time.

Needle uses Tree-sitter grammars for AST-aware chunking across 8 languages:

  • Rust — functions, structs, impls, traits
  • Python — functions, classes, methods
  • TypeScript / JavaScript — functions, classes, arrow functions
  • Go — functions, types, interfaces
  • Java — classes, methods
  • C / C++ — functions, struct definitions
  • Markdown — section-by-section prose chunking

PDF files are also supported — they're text-extracted and chunked by paragraph automatically. Any other file type falls back to sliding-window line chunking.

MCP (Model Context Protocol) is a standard that lets AI tools call external tools. Running needle mcp starts a stdio MCP server that exposes 11 tools to any compatible AI client.

For Claude Code: claude mcp add needle needle mcp

For Cursor / Windsurf: add {"mcpServers": {"needle": {"command": "needle", "args": ["mcp"]}}} to .cursor/mcp.json or .windsurf/mcp.json

For VS Code Copilot: add to .vscode/settings.json under github.copilot.chat.mcp.servers.

During indexing, Needle runs a multi-pass graph extraction over your source files:

  • Pass 1 — definition extraction: finds every function, method, class, struct, trait, and module.
  • Pass 1.5 — endpoint detection: finds Axum .route(), Express app.get(), etc. and promotes matching functions to endpoint nodes.
  • Pass 2 — call/import extraction: finds every function call, import, and use-statement, resolving them to the nodes found in Pass 1.

The resulting graph has nodes (functions, classes, endpoints, modules) and edges (calls, imports, contains). The web UI renders it with D3 force simulation. You can also export it with needle graph.

No — all search, graph, and report features work completely offline with no API keys. The Ask feature (RAG over LLM) is the only thing that needs one.

Needle checks for API keys in this priority order: ANTHROPIC_API_KEY → OPENAI_API_KEY → GROQ_API_KEY → local Ollama (no key needed). If none are set, the Ask tab shows a message to configure one.

Needle is designed for codebases of 10k–500k chunks (roughly 1,000–50,000 source files). Query latency stays under 50ms (p50) because both indexes are memory-mapped and never hit disk during search.

Memory usage: roughly 2–3 MB per 1,000 chunks for the HNSW index, plus ~300 KB per 1,000 chunks for BM25 postings. A 100k-chunk codebase uses ~300 MB RAM total — well within modern laptops.

A god node is a symbol (function, module, class) with unusually high degree in the call graph — many other things depend on it. In Needle's own codebase, graph/mod.rs has degree 51, meaning 51 other nodes call or import it directly.

God nodes are architectural load-bearers: changing them has the widest blast radius. They're the first thing to review when refactoring, and the last thing you want as a cyclic dependency. needle report ranks them by degree so you can see them at a glance.

    Privacy Policy

    Effective date: June 2025 · Needle is a local-only tool.

    The short version

    Needle collects nothing. Your code never leaves your machine. There are no analytics, no telemetry, no accounts, and no servers controlled by the Needle project that your data touches.

    What data does Needle access?

    When you run needle init, Needle reads the source files in the directories you specify. It:

    • Chunks each file into smaller pieces
    • Generates embeddings for each chunk (locally, via Ollama or hash-projection)
    • Stores the chunks, embeddings, and inverted index in a local directory on your disk (default: ~/.needle/index/)

    None of this data is transmitted anywhere.

    Ollama (optional)

    If you have Ollama running locally, Needle sends text to the local Ollama API at http://localhost:11434 to generate embeddings and LLM responses. This is a local HTTP call — data does not leave your machine.

    Groq (optional fallback)

    If you set a GROQ_API_KEY environment variable and Ollama is unavailable, the Ask feature will send your question and retrieved code chunks to the Groq API. In this case, that data is subject to Groq's privacy policy. This fallback is optional and requires you to explicitly provide an API key.

    Browser storage

    The Needle web UI stores your preferences (theme, search defaults, Ollama URL) in your browser's localStorage. This data is stored locally on your device and is not transmitted anywhere.

    Telemetry

    Needle collects no usage analytics, crash reports, or telemetry of any kind. There is no tracking code in the binary or the UI.
